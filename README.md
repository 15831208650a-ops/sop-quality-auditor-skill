# SOP Quality Auditor V11 使用说明

这是一个用于审核审计底稿 SOP 质量的 Skill。它适用于 Markdown 格式 SOP，尤其适合长文 SOP、8 万字以上 SOP、固定资产 SOP 或其他需要结构扫描、分块打分、RC 覆盖检查和可定位扣分证据的审计底稿 SOP。

## 一、部署方式

### 方式 1：作为完整 Skill 部署

适用于支持 Skill 目录结构的平台或工具。

1. 下载或克隆本仓库。
2. 将整个目录作为 Skill 使用：

```text
sop-quality-auditor-skill/
```

3. 确认目录内至少包含：

```text
SKILL.md
references/scoring_rules.md
scripts/sop_precheck.py
single-file-deploy.md
```

4. 使用时让模型调用该 Skill，并上传或提供需要审核的 SOP Markdown 文件。

### 方式 2：单文件部署

适用于 Kimi、网页聊天、知识库提示词、只支持粘贴提示词的平台。

直接使用：

```text
single-file-deploy.md
```

把 `single-file-deploy.md` 的完整内容作为系统提示词、知识库规则或自定义助手指令，再上传待审核 SOP 文件即可。

### 方式 3：打包发送给别人

如果要发给同事，建议直接发 GitHub 仓库链接，或下载整个仓库压缩包。

优先使用完整 Skill 目录；如果对方平台不支持目录结构，就发 `single-file-deploy.md`。

## 二、怎么使用

推荐提问方式：

```text
请使用 SOP Quality Auditor V11 审核我上传的 SOP，按 100 分制输出质量审核报告，要求先结构扫描，再分块审核，最后汇总评分。
```

如果平台支持文件生成，要求模型生成 Markdown 报告文件：

```text
请生成 Markdown 格式审核报告，并给出报告文件路径。
```

如果平台只能聊天输出，就要求：

```text
请直接在对话中输出完整审核报告。
```

## 三、预检脚本

如果运行环境支持 Python，可以先运行结构预检脚本：

```bash
python scripts/sop_precheck.py path/to/SOP.md --format markdown
```

脚本只做预检，不代替最终评分。它主要帮助识别：

- 必备模块是否缺失。
- PSP 是否缺少基础操作指引。
- 注释、TODO、待补充等草稿标记。
- 需拆分长句候选。
- 编号层级倒挂候选。
- GAAP Difference 是否为空表或缺少有效数据行。
- GAAP Difference、Review Checklist、常用网站附近是否有表格。

最终得分仍必须按 `references/scoring_rules.md` 或 `single-file-deploy.md` 中的规则人工分块审核后汇总。

## 四、核心打分逻辑

总分 100 分，六大板块：

| 板块 | 满分 |
|------|------|
| 基础操作指引 | 40 |
| 进阶实操提示 | 20 |
| 易错点 | 20 |
| GAAP Difference | 5 |
| Review Checklist | 12 |
| 常用网站 | 3 |

审核必须按这个顺序执行：

1. 结构扫描：识别标题、PSP、必备模块、表格、草稿标记和明显结构问题。
2. 分块审核：按六大板块逐项扣分，每条扣分必须有位置、原文短摘录、规则和分值。
3. 汇总评分：按板块汇总扣分，计算总分和结论。

结论档位：

| 得分 | 结论 |
|------|------|
| ＜30 | 立即重写 |
| 30-50 | 大段修改 |
| 50-60 | 重点修改 |
| 60-85 | 小幅修改 |
| 85-100 | 审核通过 |

## 五、重点规则

0 容忍问题会导致结论必须标注“必须修改”：

- 缺少任一必备章节。
- 标题层级严重错误，导致 PSP 章节无法识别、核心步骤层级混乱或模块归属不清。
- 同层级编号规则大量混用。
- 编号层级倒挂：标准是 `①②③` 下可以挂 `a. b. c.`，反过来 `a. b. c.` 下挂 `①②③` 属于严重结构问题。

长句不再按全文短句率抵消：

- 基础操作指引按每个 PSP 局部审核。
- 进阶实操提示按每个进阶主题局部审核。
- 重点识别一个句子或列表项里塞入多个操作、判断、条件、留痕/复核要求的情况。

GAAP Difference 不只检查列头：

- 必须有标准表格。
- 至少包含“分类 / 差异项 / PRC / HKFRS”核心列。
- 至少存在 1 行有效数据，且“差异项 / PRC / HKFRS”三列中至少两列有非空、非占位内容。
- 只有标题、列头、行头、空内容或 `-`、`/`、`N/A`、`待补充`、`同上` 等占位内容，本板块 0 分。

## 六、输出要求

审核报告至少包含：

- 基本信息。
- 各板块得分概览。
- 关键扣分明细。
- Review Checklist 覆盖检查摘要。
- 优先整改建议。

扣分必须可定位、可复核、可解释。不要用“部分”“若干”“个别”等模糊表述替代具体证据。
